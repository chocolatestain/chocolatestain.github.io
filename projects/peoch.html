겪었던 어려움 및 해결 과정

풀스택 개발 방식에서 오는 일관성 유지 문제

프로젝트를 진행하며 가장 큰 어려움 중 하나는 풀스택 개발 방식에서 오는 일관성 유지 문제였습니다. 프론트엔드와 백엔드를 기능별로 나누어 개발하다 보니, 다음과 같은 부분에서 개발 표준을 지키는 데 많은 노력이 필요했습니다.

  • 백엔드: 코딩 컨벤션, 디렉토리 구조, API 호출 방식 등을 통일하는 데 어려움을 겪었습니다. 각 기능마다 독립적으로 개발되면서 자칫 중복되거나 일관성 없는 코드가 생성될 위험이 있었습니다.
  • 프론트엔드: 사용 라이브러리, 클래스 이름 지정 규칙, 디자인 가이드라인 등을 통일하여 UI/UX의 일관성을 유지하는 것이 중요했지만, 개별 기능 개발 시 이러한 규칙을 철저히 지키는 데 어려움이 있었습니다.

이러한 문제를 해결하기 위해 사전에 명확한 개발 표준을 수립하고, 주기적인 코드 리뷰를 통해 이를 준수하도록 노력했습니다.

기능별 개발 과정에서 파트 간 정보 전달의 어려움

다음으로, 기능별 개발 과정에서 파트 간 정보 전달의 어려움이 있었습니다. 백엔드를 기능 단위로 분리하여 개발하다 보니, 각 기능에 필요한 데이터 모델 설계 및 정보 흐름을 정의하는 데 있어 협업의 중요성을 절감했습니다.

  • DTO(Data Transfer Object) 설계: 각 기능에서 필요한 데이터 구조를 정의하고 이를 프론트엔드와 공유하는 과정에서 DTO 설계의 일관성과 효율성을 확보하는 것이 중요했습니다.
  • ERD(Entity-Relationship Diagram) 설계: 데이터베이스 테이블 간의 관계를 명확히 정의하고, 기능별 요구사항을 반영하는 데 있어 초기 ERD 설계의 중요성과 변경 관리의 어려움을 경험했습니다.
  • 정보 흐름 설계: 사용자 요청부터 데이터 처리, 응답까지 이어지는 전반적인 정보의 흐름을 기능별로 명확하게 정의하고 공유하는 데 어려움이 있었습니다.

이러한 문제점을 극복하기 위해 초기 설계 단계에서 팀원들과 밀접하게 소통하며 ERD와 API 명세서를 상세하게 작성했습니다. 또한, 주기적인 스크럼 회의를 통해 각 파트의 진행 상황을 공유하고, DTO 및 정보 흐름에 대한 변경 사항을 즉시 반영하여 혼선을 줄였습니다. Jenkins를 활용한 지속적인 통합 및 배포 환경을 구축하여 변경 사항을 빠르게 공유하고 검증할 수 있도록 지원했습니다.

인프라 및 배포 단계에서의 도전과 해결

인프라 및 배포 단계에서의 도전과 해결

프로젝트 초기에는 로컬 환경에서 개발을 진행하며 localhost 기반으로 프론트엔드, 백엔드, 데이터베이스 간의 통신을 설정했습니다. 하지만 프로젝트 중간에 외부 기관에서 실제 리눅스 서버를 사용할 기회가 생기면서, 서버 배포 환경에 맞는 인프라 구성과 데이터 소통 방식 변경이라는 예상치 못한 과제에 직면했습니다.

이로 인해 다음과 같은 기술적 난관들을 해결해야 했습니다.

  • 컨테이너 간 통신 및 네트워크 설정: 로컬 환경과 달리 실제 서버에서는 여러 컨테이너(Front, Back, DB) 간의 통신 방식을 새롭게 정의해야 했습니다.
  • Reverse Proxy 및 CORS 문제 해결: 특히 프론트엔드와 백엔드 간의 API 호출 시 CORS(Cross-Origin Resource Sharing) 문제가 발생하여, Nginx를 Reverse Proxy로 활용하여 이를 해결하는 방안을 모색해야 했습니다.
  • API 호출 방식의 일관성 확보: 기존에 개발자들이 각기 다른 방식(fetch, axios, 하드코딩된 주소, 인스턴스 사용 등)으로 API를 호출하고 있어, 배포 환경에 맞춰 일관된 API 호출 규칙을 재정립하는 것이 시급했습니다. 이는 촉박한 일정으로 인해 초기에는 충분히 확립되지 못했던 부분이었습니다.
  • Pod 설계 및 배포 전략: 컨테이너 오케스트레이션 환경에서 효율적인 서비스 구동을 위한 Pod 설계가 필요했습니다.

이러한 문제들을 해결하기 위해 다음과 같은 노력을 기울였습니다.

CI/CD 구축 과정에서의 고민과 개선

CI/CD(지속적 통합/지속적 배포) 환경은 Jenkins를 활용하여 구축했습니다. 초기에는 Nginx, Spring Boot, 데이터베이스 컨테이너를 하나의 Pod으로 묶고, Jenkins 컨테이너는 배포의 역할만을 수행한다는 판단 하에 별도의 Pod으로 분리하여 관리하려 했습니다.

그러나 이 과정에서 Jenkins와 다른 서비스 Pod 간의 CORS 문제가 발생했습니다. Jenkins 구버전에서는 CORS 해제 옵션을 지원했지만, 사용 중인 신버전에서는 해당 기능이 지원되지 않아 어려움을 겪었습니다. 결국, 이 문제를 해결하고 안정적인 배포 환경을 구축하기 위해 Jenkins 컨테이너를 다른 서비스 컨테이너들과 동일한 Pod 안에 포함하여 배포를 진행하는 방향으로 선회했습니다. (향후 Jenkins와 배포 대상 서비스의 분리된 관리에 대한 추가적인 연구와 더 나은 해결 방안 모색이 필요하다고 생각합니다.)

이러한 경험을 통해 실제 배포 환경에서의 인프라 구성과 CI/CD 파이프라인 구축의 복잡성을 이해하고, 예기치 못한 문제 발생 시 유연하게 대응하고 해결 방안을 찾아나가는 능력을 기를 수 있었습니다.

성과 및 배운 점

안정적인 백엔드 시스템을 성공적으로 구축하여 프론트엔드 개발팀과의 협업을 원활하게 진행할 수 있었습니다. Spring Boot의 다양한 기능을 깊이 이해하고, CI/CD 파이프라인 구축 경험을 통해 개발 생산성을 향상시키는 방법을 배웠습니다.
